## Context

当前配置：
- Tauri 窗口：400x400，无边框透明
- PixiJS Application：400x400，canvas 通过 CSS 100% 填充视口
- 模型缩放：`Math.min(targetWidth/modelWidth, targetHeight/modelHeight) * 0.9`，填充 90% 窗口
- 悬浮菜单：6 按钮，100px 半径环形布局，44x44px 按钮

目标：缩小窗口和模型，按钮保持不变。

## Goals / Non-Goals

**Goals:**
- 窗口缩小到 300x300（减少 25% 面积）
- 模型在窗口中占比更小，视觉上更精致
- 悬浮按钮大小和布局完全不变

**Non-Goals:**
- 不实现窗口动态缩放/可调大小
- 不修改按钮半径、按钮尺寸或菜单布局
- 不修改 CSS 布局逻辑（仍用 100vw/100vh）

## Decisions

### 1. 窗口目标尺寸：300x300
**选择**: 300x300（原 400x400 的 75%）
**理由**: 按钮在 100px 半径、44x44 尺寸下，在 300 窗口中边缘留白约 28px，仍安全可见。更小的窗口（如 250）会使按钮过于拥挤。
**替代方案**: 280x280（留白仅 18px，偏紧）、320x320（缩小效果不明显）。

### 2. 模型缩放因子：0.9 → 0.6
**选择**: 乘数从 0.9 降到 0.6
**理由**: 300 * 0.6 = 180px 模型宽度，约为窗口 60%。与 100px 半径按钮（外环直径 244px）形成合理比例——模型居中，按钮环绕在外，互不遮挡。
**替代方案**: 0.5（模型太小，细节丢失）、0.7（与按钮重叠风险）。

### 3. 同步修改 PixiJS Application 尺寸
**选择**: Application width/height 从 400 改为 300，匹配窗口大小
**理由**: PixiJS 渲染分辨率与 canvas 像素尺寸一致，避免模糊或拉伸。
**替代方案**: 保持 400x400 渲染、CSS 缩放 — 浪费 GPU 资源且可能模糊。

## Risks / Trade-offs

- **[按钮相对变大]** → 按钮 44x44 在 300 窗口中占比更大，但这是预期效果，不修改按钮。
- **[模型细节缩小]** → 模型物理尺寸变小，Live2D 细节仍可辨识，因为 Live2D 是矢量渲染。
- **[窗口位置记忆]** → 已保存的窗口位置基于左上角，窗口缩小后位置会偏移。用户需重新拖拽定位。可接受。
